# S12 — H3 Security & Authentication

**Status:** Spec
**Version:** 1.0.0
**Last Updated:** 2026-07-21

---

## 1. Overview

The H3 protocol currently has zero authentication. Any process on the network can call `/v1/process` and impersonate a harness. Any harness accepts requests from any caller. This spec defines the authentication model that Secures the Hermes ↔ harness channel.

**Design principle:** Defense in depth. Three independent layers — API key (application), mTLS (transport), rate limiting (abuse prevention). Each layer can fail independently without compromising the others.

**Scope:** This spec covers the design. Implementation tasks (SEC-02 through SEC-07) are tracked on the task board. Rate limiting (Layer 3) is specified in S15.

---

## 2. Authentication Model

Three-layer security architecture:

```
┌──────────────────────────────────────────────────┐
│ LAYER 1: API Key Auth (Application)              │
│ Bearer token on every request.                   │
│ Hermes presents Hermes key.                      │
│ Harness validates caller identity.               │
├──────────────────────────────────────────────────┤
│ LAYER 2: Mutual TLS (Transport)                  │
│ Both sides present X.509 certificates.           │
│ Hermes CA signs harness certs during install.    │
│ Prevents MITM, eavesdropping, impersonation.     │
├──────────────────────────────────────────────────┤
│ LAYER 3: Rate Limiting (Abuse Prevention)        │
│ Per-harness: max decisions/sec, burst allowance. │
│ Per-session: max turns, max cost.                │
│ Global: Hermes-wide rate cap.                    │
└──────────────────────────────────────────────────┘
```

### Key Hierarchy

| Key Type | Generated By | Stored In | Scope |
|---|---|---|---|
| Harness API Key | Harness (`hermes h3 scaffold`) | `.h3-harness.yaml` (harness), `.hermes/h3/harnesses/<id>.yaml` (Hermes) | One per harness instance |
| Hermes Identity Token | Hermes (`hermes h3 install`) | `.hermes/h3/identity.yaml` (Hermes) | One per Hermes installation |
| mTLS CA Certificate | Hermes (first `h3 install`) | `.hermes/h3/ca/` (Hermes) | Signs all harness certs |
| Harness Certificate | Hermes CA | `.h3-harness.yaml` (harness) | One per harness, signed by Hermes CA |

## 3. API Key Lifecycle

### 3.1 Key Format

```
h3_<base64url(24 random bytes)>
```

Example: `h3_kxDRTqP9m2Vj7wNcY4BfL8HsA1Qe3ZpW`

- Prefix `h3_` for easy identification in logs and configs
- 24 random bytes = 192 bits of entropy
- Base64url-encoded (no special chars, safe for env vars and HTTP headers)
- Total length: 35 characters

### 3.2 Generation

**Harness side (`hermes h3 scaffold`):**

```
$ hermes h3 scaffold --lang go
  ✓ Created harness skeleton in ./my-harness/
  ✓ Generated API key: h3_kxDRTqP9m2Vj7wNcY4BfL8HsA1Qe3ZpW
  ✓ Saved to .h3-harness.yaml

  Share this key with Hermes:
    hermes h3 install --harness-url http://localhost:9191 --api-key h3_kxDRTqP9...
```

Key is generated using `secrets.token_urlsafe(24)` (Python) / `crypto/rand` (Go) / `crypto.randomBytes` (TS). Never derived from predictable input.

### 3.3 Registration

**Hermes side (`hermes h3 install`):**

```bash
hermes h3 install --harness-url http://localhost:9191 --api-key h3_kxDRTqP9...
```

Flow:
1. Hermes stores key in `.hermes/h3/harnesses/<sha256(url)>.yaml`
2. Hermes sends `POST /v1/auth/register` with Hermes identity token
3. Harness validates Hermes token, stores the pairing
4. Pairing is complete — both sides now trust each other

### 3.4 Rotation

```bash
hermes h3 rotate-key --harness-url http://localhost:9191
```

Flow:
1. Hermes tells harness: "generate new key, keep old valid for 5 minutes"
2. Harness generates new key, returns it over the authenticated channel
3. Both old and new keys work for 5 minutes (grace period)
4. After 5 minutes, old key is rejected with `KEY_EXPIRED` error
5. Hermes updates its stored key

If rotation fails mid-way (Hermes has new key, harness still has old): retry with old key, both sides revert to old key. No session disruption.

### 3.5 Revocation

```bash
hermes h3 uninstall --harness-url http://localhost:9191
```

Flow:
1. Hermes deletes harness config from `.hermes/h3/harnesses/`
2. Hermes sends `DELETE /v1/auth/pairing` to harness
3. Harness deletes Hermes identity from its trust store
4. All active sessions for that harness are terminated
5. Subsequent requests return `UNAUTHORIZED`

### 3.6 Compromise Response

If a key is leaked:
1. **Hermes key leaked:** Attacker can impersonate Hermes to every registered harness. Rotate Hermes identity token immediately (`hermes h3 rotate-identity`). All harnesses must re-register.
2. **Harness key leaked:** Attacker can impersonate one harness to Hermes. Revoke that harness key immediately (`hermes h3 revoke --harness-url ...`). Sessions using that harness are terminated.
3. **CA key leaked:** Attacker can issue valid harness certs. Re-generate CA, re-issue all harness certs. This is catastrophic — treat like a root CA compromise.

## 4. Transport Security (mTLS)

### 4.1 Certificate Architecture

```
Hermes Root CA (self-signed, stored in .hermes/h3/ca/)
  │
  ├── Hermes Server Certificate (for Hermes listening on H3 port)
  │     CN: hermes.h3.local
  │     SAN: localhost, 127.0.0.1, hermes hostname
  │
  └── Harness Certificate (one per harness, signed by Hermes CA)
        CN: harness-<sha256(url)>.h3.local
        SAN: harness hostname
```

### 4.2 Certificate Generation

During `hermes h3 install`:

```
1. Hermes CA exists?
   ├── No → Generate Hermes Root CA (Ed25519, 10-year validity)
   │         Generate Hermes server cert (signed by CA)
   └── Yes → Use existing CA

2. Hermes generates harness cert:
   - Key: Ed25519 (fast, small, modern)
   - CN: harness-<sha256(url)>.h3.local
   - SAN: harness hostname (DNS or IP)
   - Validity: 1 year
   - Signed by Hermes CA

3. Hermes sends cert + key to harness via API key-auth'd channel:
   POST /v1/auth/certificate
   Authorization: Bearer h3_<harness-key>

4. Harness saves cert + key, enables mTLS on next restart
```

### 4.3 TLS Enforcement Modes

| Mode | Hermes → Harness | Harness → Hermes | Use Case |
|---|---|---|---|
| `strict` | mTLS required | mTLS required | Production |
| `permissive` | mTLS preferred, fallback to API-key-only | mTLS preferred | Staging |
| `none` | No TLS | No TLS | Local dev only |

Mode is set per harness in `.hermes/h3/harnesses/<id>.yaml`:

```yaml
harness_url: http://localhost:9191
api_key: h3_kxDRTqP9...
tls_mode: strict
tls_ca_cert: /home/kara/.hermes/h3/ca/ca.crt
tls_harness_cert: /home/kara/.hermes/h3/ca/harness-<id>.crt
```

### 4.4 TLS Handshake

```
HERMES                                    HARNESS
  │                                          │
  │── ClientHello ─────────────────────────►│
  │◄── ServerHello + ServerCert ────────────│
  │── ClientCert ──────────────────────────►│
  │◄── Finished ────────────────────────────│
  │                                          │
  │ Both verify:                             │
  │  - Cert signed by Hermes CA              │
  │  - CN matches expected                   │
  │  - Cert not expired                      │
  │  - Cert not in CRL                       │
  │                                          │
  │── POST /v1/process (encrypted) ────────►│
```

If TLS handshake fails: connection refused. Hermes logs the failure and falls back to native loop (RES-01).

## 5. Request Authentication

### 5.1 Hermes → Harness (Every Request)

Every request from Hermes to harness includes:

```http
POST /v1/process HTTP/1.1
Host: localhost:9191
Content-Type: application/json
Authorization: Bearer h3_hx_<64-char-hex>
H3-Hermes-Identity: hermes-main
H3-Protocol-Version: 1.1
```

**Headers:**

| Header | Required | Value | Description |
|---|---|---|---|
| `Authorization` | Yes | `Bearer h3_hx_<token>` | Hermes identity token (prefixed `h3_hx_`) |
| `H3-Hermes-Identity` | Yes | Hermes instance ID | For multi-Hermes deployments |
| `H3-Protocol-Version` | Yes | Protocol version | Already present in v1.0 |

**Harness validates:**
1. `Authorization` header present and starts with `Bearer h3_hx_`
2. Token matches one in its trust store (registered during `h3 install`)
3. Token not revoked
4. `H3-Hermes-Identity` matches the token's registered identity
5. If mTLS: client certificate CN matches the registered harness hostname

### 5.2 Authentication Flow on Every Request

```
Harness receives request
  │
  ├── LAYER 1: TLS check
  │     mTLS enabled? → Verify client cert signed by Hermes CA
  │     Fail? → 401 UNAUTHORIZED + error "TLS_CERT_INVALID"
  │
  ├── LAYER 2: API key check
  │     Authorization header present?
  │     Fail? → 401 UNAUTHORIZED + error "MISSING_AUTH_HEADER"
  │     Token format valid (h3_hx_ prefix)?
  │     Fail? → 401 UNAUTHORIZED + error "INVALID_TOKEN_FORMAT"
  │     Token in trust store?
  │     Fail? → 401 UNAUTHORIZED + error "UNKNOWN_IDENTITY"
  │     Token revoked?
  │     Fail? → 401 UNAUTHORIZED + error "TOKEN_REVOKED"
  │
  ├── LAYER 3: Rate limit check
  │     Within rate limit for this identity?
  │     Fail? → 429 TOO_MANY_REQUESTS + error "RATE_LIMITED"
  │
  └── ✓ Authenticated → process request
```

### 5.3 Hermes Identity Token Format

```
h3_hx_<64 hex chars>
```

- Prefix `h3_hx_` (Hermes eXchange)
- 64 hex chars = 32 random bytes = 256 bits
- Generated once during first `hermes h3 install`
- Stored in `.hermes/h3/identity.yaml`
- Sent on every request to every harness
- All harnesses share the same Hermes identity (single Hermes instance)

## 6. Endpoint: `POST /v1/auth/register`

New endpoint for pairing Hermes with a harness.

### Request

```http
POST /v1/auth/register HTTP/1.1
Authorization: Bearer h3_<harness-api-key>
Content-Type: application/json

{
  "hermes_identity": "hermes-main",
  "hermes_token": "h3_hx_9a7b3c..."
}
```

### Response (200)

```json
{
  "registered": true,
  "harness_id": "echo-harness-01",
  "protocol_version": "1.1",
  "tls_mode": "permissive",
  "capabilities": ["tool_call", "llm_call", "text", "wait", "delegate", "end"]
}
```

### Response (401)

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Invalid harness API key"
  }
}
```

### Harness-side processing:
1. Validate `Authorization: Bearer h3_<key>` — key must match the harness's own API key (self-auth — proves the caller knows the key)
2. Store `hermes_token` + `hermes_identity` in trust store
3. Return `registered: true` + harness metadata

## 7. Endpoint: `DELETE /v1/auth/pairing`

Revoke the Hermes ↔ harness pairing.

### Request

```http
DELETE /v1/auth/pairing HTTP/1.1
Authorization: Bearer h3_hx_<hermes-token>
```

### Response (200)

```json
{
  "unregistered": true,
  "harness_id": "echo-harness-01"
}
```

### Harness-side processing:
1. Validate Hermes token
2. Remove Hermes identity from trust store
3. Terminate all active sessions
4. Return acknowledgment

## 8. Endpoint: `POST /v1/auth/certificate`

Deliver mTLS certificate to harness.

### Request

```http
POST /v1/auth/certificate HTTP/1.1
Authorization: Bearer h3_hx_<hermes-token>
Content-Type: application/json

{
  "certificate": "-----BEGIN CERTIFICATE-----\nMIID...\n-----END CERTIFICATE-----",
  "private_key": "-----BEGIN PRIVATE KEY-----\nMC4C...\n-----END PRIVATE KEY-----",
  "ca_certificate": "-----BEGIN CERTIFICATE-----\nMIID...\n-----END CERTIFICATE-----"
}
```

### Response (200)

```json
{
  "certificate_installed": true,
  "restart_required": true,
  "valid_from": "2026-07-21T00:00:00Z",
  "valid_until": "2027-07-21T00:00:00Z"
}
```

### Harness-side processing:
1. Validate Hermes token
2. Write cert + key to `.h3-harness.yaml`
3. Enable mTLS on next restart (or hot-reload if supported)

## 9. Rate Limiting

### 9.1 Token Bucket Algorithm

Each harness has a token bucket:

| Parameter | Default | Description |
|---|---|---|
| `rate` | 10 | Decisions per second (sustained) |
| `burst` | 20 | Maximum burst (bucket capacity) |
| `per_session_rate` | 2 | Decisions per second per session |
| `max_concurrent_sessions` | 50 | Maximum active sessions per harness |

### 9.2 Rate Limit Headers

Every response includes:

```http
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 7
X-RateLimit-Reset: 1626834900
```

### 9.3 Rate Limit Response (429)

```json
{
  "error": {
    "code": "RATE_LIMITED",
    "message": "Rate limit exceeded. Retry after 3 seconds.",
    "details": {
      "limit": 10,
      "remaining": 0,
      "reset_at": "2026-07-21T00:01:03Z",
      "retry_after_seconds": 3
    }
  }
}
```

Hermes behavior on 429:
1. Back off for `retry_after_seconds`
2. Queue the pending decision
3. After 3 consecutive 429s → mark harness as degraded
4. After 10 consecutive 429s → fall back to native loop

### 9.4 Burst Allowance

Burst allows short spikes above the sustained rate. The bucket refills at `rate` tokens/second. A harness can burst up to `burst` tokens instantly, then must wait for refill.

Example: rate=10, burst=20. Harness can send 20 decisions instantly, then 10/sec thereafter.

## 10. Secret Handling

### 10.1 Storage

| Secret | Location | Permissions | Format |
|---|---|---|---|
| Hermes identity token | `.hermes/h3/identity.yaml` | 0600 | YAML |
| Harness API keys | `.hermes/h3/harnesses/<id>.yaml` | 0600 | YAML |
| CA private key | `.hermes/h3/ca/ca.key` | 0400 | PEM |
| Harness API key (harness side) | `.h3-harness.yaml` | 0600 | YAML |
| Harness TLS key | `.h3-harness.yaml` | 0600 | YAML (inline PEM) |

### 10.2 Secrets in Logs

**Absolute prohibition.** No secret (API key, token, private key) may appear in:
- Log output
- Error messages
- HTTP response bodies
- Debug output
- Test output

**Redaction:** If a secret would appear in an error, replace with `<REDACTED>`:

```
Error: unauthorized — token h3_hx_<REDACTED> not in trust store
```

### 10.3 Environment Variable Override

All secrets can be provided via environment variables (for CI/CD, containers):

| Env Var | Overrides |
|---|---|
| `H3_HERMES_TOKEN` | Hermes identity token |
| `H3_HARNESS_API_KEY` | Default harness API key |
| `H3_TLS_CA_CERT` | Path to CA certificate |
| `H3_TLS_CLIENT_CERT` | Path to client certificate |
| `H3_TLS_CLIENT_KEY` | Path to client key |

Env vars take precedence over config files. If both are present, env var wins.

### 10.4 Secret Audit

`hermes h3 verify` checks:
1. All secret files have correct permissions (0600/0400)
2. No secrets in git-tracked files
3. No secrets in environment (env vars are OK, hardcoded in source is NOT)
4. CA private key exists and is readable only by owner
5. Harness API key entropy ≥ 192 bits

## 11. Error Codes (New)

| Code | HTTP | Meaning |
|---|---|---|
| `UNAUTHORIZED` | 401 | Missing or invalid auth (umbrella for all auth failures) |
| `MISSING_AUTH_HEADER` | 401 | No `Authorization` header present |
| `INVALID_TOKEN_FORMAT` | 401 | Token does not match expected format |
| `UNKNOWN_IDENTITY` | 401 | Token not in trust store |
| `TOKEN_REVOKED` | 401 | Token was explicitly revoked |
| `TLS_CERT_INVALID` | 401 | Client certificate validation failed |
| `TLS_REQUIRED` | 426 | Harness requires mTLS but request used plain HTTP |
| `RATE_LIMITED` | 429 | Rate limit exceeded |
| `KEY_EXPIRED` | 401 | API key is past its grace period after rotation |

All auth errors return the standard error response shape:

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Human-readable description",
    "details": {}
  }
}
```

```
  │                                    │

  │                                    │

  │                     │              │
  │── POST /v1/result ───────────────►│
  │                                    │
  │◄── Decision {type: "end"} ────────│
  │                                    │
  ▼                                    ▼

```

## 13. Backward Compatibility

### 13.1 Protocol Version Negotiation

Current protocol version: `1.0` (no auth). Auth introduces `1.1`.

During health check:
```json
{
  "status": "ok",
  "protocol_version": "1.1",
  "auth_required": true,
  "auth_methods": ["bearer_token", "mtls"]
}
```

### 13.2 Migration Path

1. Hermes upgraded to v1.1 → sends `H3-Protocol-Version: 1.1` header
2. Harness v1.0 ignores unknown headers, doesn't check auth → still works (permissive)
3. Harness v1.1 checks auth → rejects if missing → Hermes falls back to native
4. User upgrades harness to v1.1 → re-runs `hermes h3 install --api-key ...`
5. After all harnesses upgraded → set `tls_mode: strict` globally

### 13.3 Grace Period

During migration, Hermes supports both `1.0` (no auth) and `1.1` (auth). Harnesses advertising `protocol_version: "1.0"` in health check are treated as legacy — auth not enforced, but logged with warning.

## 14. Security Considerations

### Threat Model

| Threat | Mitigation |
|---|---|
| Attacker on same network impersonates harness | API key + mTLS — both must match |
| Attacker replays captured requests | TLS prevents replay; `decision_id` uniqueness prevents double-execution |
| Attacker obtains API key from logs | Redaction + 0600 permissions + env var support for containers |
| Attacker compromises harness host → steals key | Key rotation. Old key revoked within 5 minutes. |
| DoS via request flood | Rate limiting per harness + per session. Circuit breaker (RES-03). |
| Downgrade attack (force TLS→plain) | `tls_mode: strict` prevents downgrade. Hermes refuses plain connections. |
| CA private key compromise | Re-issue all certs. Catastrophic but contained — only affects H3, not Hermes itself. |

### Key Rotation Cadence

| Key | Rotation | Rationale |
|---|---|---|
| Hermes identity token | 90 days | Long-lived but rotated regularly |
| Harness API key | 30 days | Shorter — harnesses are less trusted |
| CA certificate | 1 year | Standard CA rotation |
| Harness TLS cert | 90 days | Auto-renewed by Hermes |
