1. Executive Summary

One protocol. Every platform. Any language.

TotalStack is the universal interface between AI agent platforms and the agent harnesses that think. It replaces tightly-coupled, vendor-locked architectures with a clean, two-endpoint contract. The platform handles infrastructure — messaging, tools, models, credentials, scheduling, plugins. The harness handles intelligence — reasoning, planning, memory, decision-making. Any harness that implements TotalStack instantly gains everything the platform offers, without writing a single line of platform code.

"The platform asks: 'What should I do?' The harness answers. The platform executes. The platform reports back. Loop."

The harness thinks. The platform does. That's the split.

🔒 The Problem

Agent harnesses are tightly coupled to platforms. Switching platforms means rewriting the harness. Every new project re-implements the same integration layer. Harness developers spend 70% of their time on infrastructure plumbing instead of agent logic.

🔑 The Solution

A thin, two-endpoint contract. Implement it once. Polyglot SDKs for Go, Python, and TypeScript — with code generation for any language. A 43-test compliance battery gates every harness. The platform evolves independently from the brain.

3. Protocol Specification

OpenAPI 3.1 single source of truth — 6 endpoints, 6 decision types

Endpoints

MethodEndpointPurposeRequired
GET /v1/health Health check — status, version, capabilities, latency YES
POST /v1/process Core: New message → harness returns a Decision YES
POST /v1/result Core: Execution outcome → harness returns next Decision YES
POST /v1/cancel Cancel in-flight operation (user interrupt) Recommended
GET /v1/sessions/:id Session metadata — turn count, status, active decision Optional
DELETE /v1/sessions/:id Force session termination Optional

What the Platform Sends to Every Harness

FieldDescription
session_idStable session identifier across turns
messageUser message — content, attachments, timestamp
identityWho sent it — platform, user ID, chat ID, thread ID
context.historyLast N messages (configurable, default 20)
context.toolsAvailable tools with full JSON Schema params
context.modelsAvailable LLMs with pricing, context window, capabilities
context.memoryPersistent memory relevant to this session
context.configSession config — max iterations, timeout, temperature

Decision Types

The harness returns exactly one Decision per response. The platform executes it and reports back.

tool_call
Execute a platform tool — terminal, web search, file I/O. Harness picks the tool and params. Platform sandboxes execution.
→ Platform executes → POST /v1/result
llm_call
Route a prompt to an LLM. Harness selects from available models. Platform handles provider routing and cost tracking.
→ Platform routes to model → POST /v1/result
text
Send Markdown-formatted text to the user. Set finished: false to continue the loop. finished: true to end the exchange.
finished: false → continue loop
wait
Pause for an external signal — CI completion, human approval, API callback. Optional polling endpoint.
→ timeout / signal → POST /v1/result
delegate
Spawn a sub-agent for parallel work. Platform manages the sub-agent lifecycle. Harness gets the summary.
→ sub-agent complete → POST /v1/result
end
Terminate the session. Six reasons: task_complete, user_requested, error, timeout, rate_limited, cancelled.
→ session terminates cleanly

Session Lifecycle

1. User message arrives (Telegram / Discord / Slack / Web)
2. Platform authenticates, rate-limits, resolves session
3. Shim → POST /v1/process (session, message, tools, models, history, memory)
4. Harness thinks → returns Decision
5. Platform executes the Decision
6. Platform → POST /v1/result (execution outcome)
7. Harness thinks → returns next Decision
8. Loop until harness returns end
9. Response delivered to user

4. Multi-Language SDKs

Three first-party SDKs generated from a single OpenAPI 3.1 spec — idiomatic bindings, not mechanical format translation.

Go SDK

Strict types with JSON tags. Harness interface. Built-in http.Handler server with middleware (logging, recovery, timeout). Mock platform for testing. One line to serve: NewHTTPServer(harness).

Python SDK

Pydantic v2 models. BaseHarness ABC with async methods. FastAPI router for one-line integration. Mock platform + pytest fixtures. Published on PyPI.

TypeScript SDK

Zod schemas for runtime validation. Harness interface. Hono router for quick-start server. Mock platform + vitest helpers. Published on npm.

Minimal Harness Example (Go — ~25 lines)

// Echo harness — the simplest possible TotalStack harness
type EchoHarness struct{}

func (e *EchoHarness) OnProcess(req *protocol.ProcessRequest) (*protocol.Decision, error) {
    return &protocol.Decision{
        Decision:  protocol.DecisionText,
        Text: &protocol.TextResp{
            Content:  fmt.Sprintf("You said: %s", req.Message.Content),
            Finished: true,
        },
    }, nil
}

func (e *EchoHarness) OnResult(req *protocol.ResultRequest) (*protocol.Decision, error) {
    return &protocol.Decision{Decision: protocol.DecisionEnd, End: &protocol.End{Reason: "task_complete"}}, nil
}

func (e *EchoHarness) Health() *protocol.HealthResponse {
    return &protocol.HealthResponse{Status: "ok", Version: "1.0.0", Transport: "rest"}
}

// One line to serve:
http.ListenAndServe(":9191", harness.NewHTTPServer(&EchoHarness{}))

Zero → working harness in under 5 minutes. Implement three methods. Run the compliance test. Done. The SDK handles JSON marshaling, validation, middleware, error recovery, and session management.

6. Security Model

Security by architecture, not by configuration.

PropertyGuarantee
Network boundaryHarnesses bind to localhost only — no network exposure by default
AuthenticationPlatform authenticates the user; harness trusts the platform
Credential isolationHarness cannot access the platform credential vault directly
Tool sandboxingHarness cannot bypass tool execution sandboxing
Audit trailAll tool execution is logged and attributed to the harness session
AuthorizationUser identity passed in context.identity — harness decides policy

7. Error Handling & Resilience

Graceful degradation at every failure point.

Failure ModeBehavior
Harness unreachable at startupLog warning, route all sessions to platform native fallback
Harness timeout mid-sessionRetry 3× with exponential backoff (1s, 2s, 4s)
Malformed Decision returnedLog error, return error to user, end session
Harness hangs (no response)Timeout at configured timeout_ms, terminate session
Health check fails (3× consecutive)Remove from pool, poll every 30s, restore on success
Harness returns end with errorRelay error to user, terminate cleanly

10. Design Principles

🧠 Harness Thinks, Platform Does

Clean separation of concerns. The harness never touches files, manages API keys, or configures gateways. It thinks. The platform does.

🔌 Language-Agnostic

REST + JSON means any language works. gRPC for high-frequency loops. A curl script is a valid harness if it speaks the protocol.

🛡️ Secure by Architecture

Harnesses run on localhost. No network exposure. Cannot access credential vaults. All tool execution is sandboxed and attributed.

🧪 Compliance-Driven

The test battery IS the spec. 43 tests. Region-style E2E coverage. Any developer can run totalstack-test against their endpoint.
