Native → H3 Migration

You have a native Hermes agent. Now give it upgrade survival, language freedom, and independent deployment.

Hermes v0.18+ H3 Protocol v1.0 ~30 minute migration

What Changes

Before (Native)

Your agent logic lives inside Hermes — a Python module in hermes_cli/agent/. Every Hermes update risks breaking it. Deployment means patching the Hermes installation.

After (H3)

Your agent logic is an independent service — a Go, Python, or TypeScript process. Hermes calls it via HTTP. Upgrade Hermes freely; your harness stays untouched. Deploy anywhere.

Before (Native)
# hermes_cli/agent/my_agent.py
from hermes_core import ToolRegistry, ModelRouter

class MyAgent:
    def on_message(self, msg):
        # Direct access to Hermes internals
        tools = ToolRegistry.list()
        result = ModelRouter.call("deepseek-v4-pro", msg)
        return result
After (H3)
// harness.go — independent service
func (h *MyHarness) OnProcess(
    req protocol.ProcessRequest,
) (*protocol.Decision, error) {
    // Hermes sends context. You decide.
    // No access to internals needed.
    reply := h.agent.Think(req.Context.History)
    return &protocol.Decision{
        Type: protocol.DecisionText,
        Content: reply,
    }, nil
}

The Promise

After migration: upgrade Hermes without touching your agent. Switch languages without changing Hermes. Deploy independently — your harness is just an HTTP server. The shim handles tool/model routing, session management, and protocol compliance.

Migration Steps

1 Audit Your Native Agent

Identify what your agent actually uses. Most agents only touch a small subset of Hermes internals.

What to audit

  • Which tools does your agent call? (web_search, terminal, memory, etc.)
  • Which models does it request?
  • Does it access session state? (context.history, context.messages)
  • Does it call external APIs?
  • Does it use Hermes-specific features? (plugin hooks, internal caches, direct DB access)
Common surprise: Most agents only use 3–5 tools and a single model. The migration is usually simpler than it looks.

Hermes-specific features that need workarounds

Feature Native Access H3 Equivalent
Tool calls ToolRegistry.call(name, args) Return DecisionToolCall — shim executes
LLM calls ModelRouter.call(model, prompt) Call your own LLM API directly, or return DecisionLLMCall
Session state context.history req.Context.History in every request
Hermes config read config.get("key") Pass config to harness at startup (env vars or config file)
Plugin hooks Direct hook registration Not supported yet — keep minimal native shim if needed
2 Pick Your SDK

Choose the language that matches your team. All three SDKs implement the same protocol.

SDK Install Best For
Go go get github.com/get-h3/sdk-go Low-latency, single binary, existing Go infra
Python pip install h3-harness-sdk Quick prototypes, ML/AI, existing Python code
TypeScript npm install @get-h3/h3-harness-sdk Web-native, Node.js, existing TS/JS infra
3 Implement the Harness Interface

All H3 harnesses implement the same three-method interface. Port your agent logic into OnProcess.

The interface (Go example)

type Harness interface {
    Health() (*protocol.HealthResponse, error)
    OnProcess(req protocol.ProcessRequest) (*protocol.Decision, error)
    OnResult(req protocol.ResultRequest) (*protocol.Decision, error)
}

Migration by agent type

Simple reply bot

Your agent reads a message and returns text. This is the echo harness with your logic swapped in. 5 minutes.

Tool-using agent

Your agent calls Hermes tools. Return DecisionToolCall objects. The shim executes them and sends results back via OnResult. Most of your logic moves to OnResult — inspect tool output, decide next action.

Multi-turn conversational agent

Your agent maintains conversation state across turns. Use req.Context.History for past turns and req.Config for runtime settings. Set Finished: false to keep the conversation open.

LLM-orchestrator agent

Your agent calls LLMs to make decisions. Either call the LLM API directly from your harness (your own keys, your own provider), or return DecisionLLMCall to have the shim route through Hermes' configured providers.

4 Start Your Harness

H3 harnesses are HTTP servers. Start yours on a local port.

# Go
go run ./examples/echo/

# Python
python -m your_harness.main

# TypeScript
npx tsx examples/echo.ts

# Output: H3 harness listening on http://localhost:9191

Verify the health endpoint:

curl http://localhost:9191/health
# {"status":"ok","version":"1.0.0","capabilities":["text","tool_call","llm_call"]}
5 Configure Hermes

Point Hermes at your harness. Edit ~/.hermes/profiles/default/config.yaml:

harnesses:
  _schema: 1
  my-agent:
    endpoint: http://localhost:9191
    transport: rest

sessions:
  "telegram:YOUR_USER_ID":
    harness: my-agent

Restart Hermes and message yourself. Your harness handles it.

6 Run the Test Battery

The H3 test battery validates 43 protocol behaviors. Run it against your harness to catch edge cases.

pip install hermes-h3-shim
h3-test --endpoint http://localhost:9191

Aim for 43/43 before going live. Each gap is a production edge case waiting to surface. The test battery runs in <5 seconds and catches protocol-level issues that manual testing misses.
7 Cut Over (Gradual)

Don't big-bang. Use this sequence:

  1. Shadow mode: Configure both native and H3 harness. Route a subset of sessions to H3. Monitor for 24 hours.
  2. Ramp up: Move 50% of sessions to H3. Compare responses side-by-side (log both). Fix discrepancies.
  3. Full cutover: All sessions on H3. Keep native as fallback for 48 hours.
  4. Archive native: Remove the native module. Clean up config.

At each stage, verify with h3-test --endpoint http://localhost:9191 and check Hermes logs for errors.

What You Get: Upgrade Survival

Before Hermes updates

hermes update  # → H3 pre-flight hook runs
# Checks: protocol compatibility, H3 version, harness health, config schema
# Result: OK → proceed, WARN → proceed with caveats, BLOCK → stop

After Hermes updates

hermes h3 verify
# Checks: shim imports OK, plugin registered, harnesses healthy,
#         test battery passes, config schema migrated, version matrix OK

If something breaks

hermes downgrade --version 0.18.0
cp ~/.hermes/backups/h3-config-*.yaml ~/.hermes/profiles/default/config.yaml
hermes h3 verify

H3 configs are automatically backed up to ~/.hermes/backups/ before every hermes update and every hermes h3 install.

Common Pitfalls

Tool name drift

Hermes renames tools between versions. The shim maps legacy names:

# Use "web_search" in your harness Decisions
# Shim maps: "search" → "web_search", "browser_open" → "browser_navigate"
# Removed tools return clear errors: "execute_code: Removed. Use terminal() instead."

Model provider changes

If a model your harness requests is unavailable, the shim auto-falls-back:

# deepseek-v4-pro unavailable → falls back to glm-5.2
# Your harness gets a warning log, not a crash

State assumptions

Don't store state in memory across requests. The shim may route to different harness instances. Use req.Context.History for conversation state and external storage (DB, Redis) for persistent state.

Native-only features

Plugin hooks and direct Hermes DB access have no H3 equivalent. If your agent depends on these, either:

  • Refactor to use HTTP-accessible Hermes APIs
  • Keep a thin native shim that proxies these features over HTTP to your harness
