1 Executive Summary

MusterFlow is the open protocol layer between AI platforms and autonomous agent harnesses. It defines a stable, language-agnostic contract that lets teams build custom AI agent brains in any programming language, wire them into the Hermes ecosystem, and immediately gain access to: multi-platform messaging gateways (Telegram, Discord, Slack, Signal, Web), a full tool registry with sandboxed execution, model routing across 8+ LLM providers, credential vaults, cron scheduling, MCP server management, a plugin system, and persistent memory — without writing a single line of platform infrastructure code.

The project is built around one design principle:

"The platform asks: 'What should I do?' The harness answers. The platform executes. The platform reports back. Loop."

The harness thinks — reasoning, planning, memory strategy, iteration logic, failure recovery, prompt engineering. The platform does — all infrastructure, I/O, and external integration. They evolve independently. They scale separately. They can be built by different teams.

2 The Problem & Solution

The Problem

Every AI agent platform today couples its reasoning loop to its infrastructure. If you need Telegram + tool sandboxing + model routing + persistent memory + cron + multi-provider fallback, you have two bad choices: (1) use the platform's built-in agent — one-size-fits-all, slow to evolve, can't customize the reasoning loop; or (2) build everything yourself — months of infrastructure work before you write your first line of actual agent logic. Neither scales. Both trap you.

The Solution

MusterFlow is three REST endpoints. Implement them, and your custom agent brain immediately inherits the full Hermes platform. The protocol is the contract. The platform is the platform. They are decoupled by design.

Session Lifecycle — End to End

Here is what happens when a user sends a message on any connected platform:

1. User message → Hermes Gateway (Telegram/Discord/Slack/Signal/Web)
2. Gateway authenticates user, resolves session → MusterFlow Shim Loader
3. Shim assembles full context → POST /v1/process
       → {session_id, message, identity, context {history, tools, models, memory, skills, config}}
4. Your harness receives context, reasons, returns a Decision
       → tool_call | llm_call | text | wait | delegate | end
5. Shim executes the Decision (runs tool, calls LLM, sends text, spawns sub-agent)
6. Execution result → POST /v1/result
       → {decision_id, result_type, outcome, data}
7. Your harness processes the result, returns next Decision
8. Loop continues until harness returns end

Key insight: The harness is stateless between turns. Every /v1/process and /v1/result call includes the full conversation history, tools, models, and session state. The harness can be a simple function — no database required unless you want one.

4 Protocol Design — The Wire Contract

MusterFlow defines six REST endpoints (with gRPC available for high-throughput use cases). The protocol is specified as OpenAPI 3.1 YAML with JSON Schema for every type — 14 schema files in total, fully validated.

MethodEndpointPurposeCritical?
GET /v1/health Health check — status, version, capabilities. Polled every 30s. Yes
POST /v1/process New message → first Decision. Full context included. Yes
POST /v1/result Execution result → next Decision. Loop backbone. Yes
POST /v1/cancel Cancel in-flight operation — user interrupt or /stop. Recommended
GET /v1/sessions/:id Session metadata — turn count, cost, uptime. Optional
DELETE /v1/sessions/:id Force session termination — cleanup, /new handling. Optional

What the Platform Sends to Your Harness

Every POST /v1/process request carries everything your harness needs to reason:

context.memorystringDuckBrain persistent memory relevant to this session

context.session_stateSessionStateRunning counters — turn count, tool calls, LLM calls, cost so far, uptime

Error Handling — Graceful Degradation

Failure ModePlatform Behavior
Harness unreachable at startupLog warning, route all sessions to native Hermes agent (zero-downtime fallback)
Harness timeout mid-sessionRetry 3× with exponential backoff (1s → 2s → 4s)
Malformed Decision returnedReturn standardized error to user, terminate session
Hang — no response within timeout_msTimeout, terminate session, log incident
Health check fails 3× consecutiveRemove harness from pool, poll every 30s for recovery

All errors use a standardized error shape with machine-readable error codes: INVALID_REQUEST, INVALID_DECISION, UNKNOWN_TOOL, UNKNOWN_MODEL, SESSION_NOT_FOUND, SESSION_EXPIRED, HARNESS_TIMEOUT, INTERNAL_ERROR.

5 Decision Types — The Agent's Vocabulary

Every harness response is exactly one Decision — a discriminated union with a decision discriminator field and a unique decision_id for tracing. The platform routes it, executes it, and reports the result back. Here are all six types:

tool_call
Execute a Hermes tool — terminal, web search, file read/write, browser automation, image generation, email, and 30+ more. Harness specifies name + params. Hermes executes in sandbox. Optional reasoning field for audit trail.
llm_call
Route a prompt to an LLM. Harness picks model from the available list, sets system_prompt, messages, and temperature. Platform handles provider routing, fallback, caching, and cost tracking.
text
Send markdown-formatted text to the user. finished: true = final message (session ends after delivery). finished: false = intermediate message (platform calls /v1/result for next Decision).
wait
Pause for an external signal. Optional poll_endpoint — Hermes polls every 15 seconds, resumes on success. Capped at timeout_seconds from session config. Enables human-in-the-loop workflows.
delegate
Spawn a sub-agent. Harness defines task, context, optional model, and toolsets. Platform manages the complete sub-agent lifecycle — spawn, run, collect result, return to harness.
end
Terminate the session. Specifies reason — one of: task_complete, user_requested, error, timeout, rate_limited, cancelled. Optional summary of what was accomplished.

The Loop in Practice

# A typical harness turn (pseudocode)

OnProcess(req):
    decision = tool_call(name="web_search", params={"query": req.message.content})
    return decision

# Platform executes web_search, returns result to:

OnResult(req):   # req.result = web search results
    decision = llm_call(model="deepseek-v4-pro", messages=[...search results...])
    return decision

# Platform runs LLM, returns response to:

OnResult(req):   # req.result = LLM response text
    decision = text(content=req.result.text, finished=true)
    return decision

# Platform delivers text to user. Session complete.

Structs, dataclasses, or interfaces for every request/response type in the protocol — generated from the OpenAPI spec, never hand-written.

Quick-start server that handles marshalling/unmarshalling, Content-Type negotiation, CORS — just register your handler functions.

Abstract class or interface with OnProcess(), OnResult(), OnCancel(), and Health() — implement these four, you're done.
✅

Request + Decision validation against the JSON Schema before anything hits the wire — catch protocol violations at build time, not in production.
🧪
Mock Test Bed
Full MockHermes client for unit testing your harness logic — simulate process calls, result returns, cancellations, and error conditions without a real platform.
📝
Examples
Two working examples per SDK — minimal (echos back) and echo (full conversation loop) — both pass the compliance test battery. Use as starting templates.

Harness Interface — Four Methods

// Implement these four methods. The SDK handles HTTP, marshalling, and validation.
type Harness interface {
    OnProcess(req *ProcessRequest) (*Decision, error)
    OnResult(req *ResultRequest) (*Decision, error)
    OnCancel(req *CancelRequest) error
    Health() *HealthResponse
}

// Start the server:
server := harness.NewServer(myHarness, ":9191")
server.ListenAndServe()

8 Security Model

🔒
Localhost-Only Binding
Harnesses bind to 127.0.0.1 by default. No network exposure unless explicitly configured. The harness and platform run on the same host.
🛡️
Platform Authenticates Users
Hermes authenticates the user at the gateway level. The harness receives identity (platform, user ID, chat ID) and decides authorization — no credential handling needed.
🔐
Credential Isolation
Harnesses cannot access the Hermes credential vault directly. All API calls go through the tool sandbox. Secrets never leave the platform.
📋
Full Audit Trail
Every tool execution, LLM call, and decision is logged with the harness session ID. All errors use standardized machine-readable codes. Complete traceability from user message to response.
