§1 Executive Summary

H3 is the universal interface between AI agent platforms and the agent harnesses that think. It replaces tightly-coupled, vendor-locked agent architectures with a clean, two-endpoint contract: the platform handles infrastructure (tools, models, credentials, scheduling, gateways), and the harness handles intelligence (reasoning, planning, memory, decision-making). Any harness implementing H3 instantly gains access to the full Hermes platform — Telegram, Discord, Slack, 100+ tools, cron, MCP, multi-model routing — without writing a single line of platform code.

The AI agent ecosystem today is fragmented. Every platform invents its own agent API. Every harness is locked to one platform. Agent developers spend 70% of their time on infrastructure plumbing instead of the agent itself. H3 fixes this.

🎯 The Problem

  • Agent harnesses are tightly coupled to platforms
  • Switching platforms means rewriting the harness
  • Every new agent project re-implements the same integration layer
  • No standard way to test harness compliance
  • Platform upgrades break harness integrations

✅ The Solution

  • A two-endpoint contract — think "CGI for AI agents"
  • Polyglot SDKs — Go, Python, TypeScript, with code generation for any language
  • A 43-test compliance battery — pass it, you're compliant
  • Hermes handles infrastructure; the harness handles intelligence only
  • Zero platform code required in harnesses

The Core Contract

Three endpoints. That's the entire spec:

EndpointMethodPurpose
/v1/healthGETHealth check + capabilities advertisement
/v1/processPOSTNew user message → agent returns a Decision
/v1/resultPOSTHermes executed the decision → what next?

Six Decision Types

The harness returns exactly ONE decision per response. Hermes executes and reports back.

DecisionPurposeExample
tool_callExecute a platform toolRun a shell command, search the web, read a file
llm_callCall an LLM"Review this Go code for security issues"
textSend text to user"Found the bug — missing JWT expiry check"
waitPause for external signalWait for CI pipeline to finish
delegateSpawn a sub-agent"Review auth.go for vulnerabilities"
endTerminate sessionTask complete, user requested stop, or error
Design Principle: "Hermes asks: 'What should I do?' The harness answers. Hermes executes. Hermes reports back. Loop." — The harness never touches files, never manages API keys, never configures gateways. It thinks. Hermes does.

Minimal Echo Harness (Go — 30 lines)

type EchoHarness struct{}

func (e *EchoHarness) OnProcess(req *protocol.ProcessRequest) (*protocol.Decision, error) {
    return &protocol.Decision{
        Decision:   protocol.DecisionText,
        DecisionID: uuid.New().String(),
        Text: &protocol.TextResp{
            Content:  fmt.Sprintf("You said: %s", req.Message.Content),
            Finished: true,
        },
    }, nil
}

func (e *EchoHarness) OnResult(req *protocol.ResultRequest) (*protocol.Decision, error) {
    return &protocol.Decision{
        Decision: protocol.DecisionEnd,
        DecisionID: uuid.New().String(),
        End: &protocol.End{Reason: "task_complete"},
    }, nil
}

func (e *EchoHarness) Health() *protocol.HealthResponse {
    return &protocol.HealthResponse{
        Status: "ok", Version: "1.0.0", Transport: "rest",
        ProtocolVersion: "1.0", Capabilities: []string{"text", "end"},
    }
}

// One line to serve:
// http.ListenAndServe(":9191", harness.NewHTTPServer(&EchoHarness{}))
Zero → working harness in under 5 minutes. Implement three methods, run h3-test, done. The SDK handles JSON marshaling, validation, middleware, error recovery, and session management.

§8 Error Modes & Resilience

FailureBehavior
Harness unreachable at startupLog warning, route to native fallback
Harness timeout mid-sessionRetry 3× with exponential backoff (1s, 2s, 4s)
Malformed DecisionLog error, return error to user, end session
Harness hangs (no response)Timeout at timeout_ms, terminate session
Health check fails (3× consecutive)Remove from pool, re-add on next success (30s polling)
end with errorRelay error to user, terminate

§11 Key Differentiators

🆚 vs. LangChain/LlamaIndex

  • Those are frameworks. H3 is a protocol.
  • No vendor lock-in — harnesses work with any platform
  • H3 harnesses don't include platform code — pure intelligence

🆚 vs. Custom Agent APIs

  • Every platform invents its own agent API. H3 is one standard
  • 43-test compliance suite — no guessing if it works
  • SDKs handle transport; harness devs write OnProcess() and OnResult()
Bottom line: H3 is the missing standard for AI agent interoperability. It's thin enough to implement in an afternoon, rigorous enough to validate with a 43-test compliance suite, and open enough to work with any platform, any language, and any agent architecture.
