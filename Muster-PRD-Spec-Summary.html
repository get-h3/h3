§1 Executive Summary

Muster is a dynamic API client that configures itself from remote inventory. Instead of installing stale SDKs or manually updating MCP packages, Muster downloads configuration from your API's _inventory.json, caches it locally, and generates commands automatically from your OpenAPI spec. It works offline, respects versioning, and integrates seamlessly with AI assistants via the Model Context Protocol (MCP).

The world of API tooling is stuck in a codegen loop. Every new API means a new SDK. Every SDK update means waiting on maintainers. Every AI agent integration means hand-writing MCP tools. Muster breaks this cycle. Point it at an OpenAPI spec and it generates everything dynamically — CLI commands, MCP tools, daemon proxies, WASM transforms, and Starlark workflow scripts. One binary. Seven surfaces. Zero codegen.

🎯 The Problem

  • API SDKs are stale the moment they're published
  • AI agents need hand-crafted MCP tools for every API
  • Multi-step API workflows require custom scripting
  • No standard way to proxy, transform, and rate-limit API traffic
  • Every API integration is a bespoke project

✅ The Solution

  • CLI — openapi-cli request GET /users with auto-completion
  • MCP Server — every OpenAPI operation becomes an MCP tool automatically
  • Daemon — Unix socket proxy with shared state, auth, and rate limiting
  • Go Library — 26 importable packages for programmatic use
  • WASM Plugins — sandboxed response transformation (wazero runtime)
  • Starlark DSL — multi-step API workflows with http_get/post/put/delete builtins
  • Docker — containerized deployment with distroless base images

§2 The Seven Surfaces

🖥️ CLI — openapi-cli

For humans. Auto-discovers operations, generates shell completions, supports auth headers, WASM transforms, and DSL execution. openapi-cli discover https://api.example.com/openapi.json

🤖 MCP Server — openapi-mcp

For AI agents. Every OpenAPI operation becomes an MCP tool. Drop it into Claude Desktop, Cursor, or any MCP client. The AI never knows it's talking to a REST API.

🔌 Daemon — openapi-daemon

For infrastructure. Unix socket proxy with multi-client support, shared state, authentication, rate limiting, and caching. curl --unix-socket /tmp/muster.sock http://localhost/users

📦 Go Library — pkg/

For other Go programs. 26+ importable packages — parser, generator, client, auth, cache, rate limiting, WASM runtime, Starlark interpreter, MCP protocol. The same packages the CLI uses.

🧩 WASM Plugins — --wasm-plugin

Sandboxed response transformation via wazero runtime. Write transforms in any language that compiles to WASM. Secure by default — each plugin runs in its own isolated sandbox.

⭐ Starlark DSL — dsl run

Multi-step API workflows in a safe, deterministic scripting language. Built-in http_get, http_post, http_put, http_delete. Chain operations, transform responses, handle errors.

🐳 Docker

Multi-stage Dockerfile with security scanning and distroless base. Docker Compose with 8 services. Kubernetes manifests, Helm charts, Debian packaging, Homebrew tap.

Core Pipeline

StagePackageWhat It Does
1. Discoverpkg/inventoryFetches _inventory.json from remote, caches locally, manages versions
2. Parsepkg/openapiParses OpenAPI 3.x spec using kin-openapi, extracts operations, parameters, schemas
3. Generatepkg/generatorConverts parsed operations into CLI commands and MCP tools with typed parameters
4. Executepkg/clientHTTP client with auth (OAuth2, mTLS, bearer), retry, rate limiting, caching
5. Transformpkg/wasmWASM sandbox (wazero) for response transformation plugins
6. Orchestratepkg/dslStarlark interpreter for multi-step workflows with built-in HTTP functions
7. Exposepkg/mcpFull MCP protocol implementation — tools/list, tools/call, resources, prompts
8. Proxypkg/daemonUnix socket proxy server with multi-client support and shared state

§13 Key Differentiators

🆚 vs. Traditional API SDKs

  • SDKs are static — Muster is dynamic, configures from remote inventory
  • SDKs require per-API installs — Muster is one binary for any API
  • SDKs have no MCP/AI integration — Muster generates MCP tools automatically
  • SDKs have no DSL — Muster has Starlark for multi-step workflows

🆚 vs. Hand-Crafted MCP Tools

  • Manual MCP tools take hours per API — Muster does it instantly
  • Manual tools go stale — Muster re-discovers on every spec change
  • Manual tools are one-off — Muster is systematic (every operation → tool)
  • Manual tools lack rate limiting, auth, caching — Muster bundles them all

🆚 vs. Postman / Insomnia / curl

  • Those are manual tools for humans — Muster is automation infrastructure
  • Muster generates commands, not collections — programmatic by design
  • MCP server means AI agents can use APIs without human intervention
  • Daemon mode provides a persistent proxy layer — not a one-off request tool
Bottom line: Muster is the API client that configures itself. Point it at an OpenAPI spec and you get a CLI, an MCP server for AI agents, a daemon proxy for infrastructure, a Go library for developers, WASM plugins for extensibility, and a Starlark DSL for workflows — all from a single static binary. No codegen. No boilerplate. No SDK hell.
