§1 Executive Summary

H3 is the missing standard for AI agent interoperability. It defines a two-endpoint contract between an AI platform (Hermes) and agent harnesses. The platform provides infrastructure — tools, models, gateways, credentials, scheduling. The harness provides intelligence — reasoning, planning, memory, decision-making. Implement two HTTP endpoints, pass a 43-test compliance suite, and your agent instantly has access to the full Hermes platform: Telegram, Discord, Slack, 100+ tools, multi-model routing, MCP servers, and cron scheduling — zero platform code required in the harness.

Today's AI agent ecosystem suffers from tight coupling. Every platform invents its own agent API. Every harness is locked to one platform. Agent developers spend 70% of their time on infrastructure plumbing instead of building the agent itself. H3 fixes this: one protocol, three SDKs, a universal compliance test, and a clean separation of concerns.

The Problem

  • Agent harnesses are tightly coupled to platforms
  • Switching platforms requires rewriting the harness
  • Every new agent project re-implements the same integration layer
  • No standard way to test harness compliance
  • Platform upgrades break harness integrations

The Solution

  • A two-endpoint contract — "CGI for AI agents"
  • Polyglot SDKs — Go, Python, TypeScript + codegen for any language
  • A 43-test compliance battery — pass it, you're compliant
  • Hermes handles infrastructure; harness handles intelligence only
  • Zero platform code required in harnesses

§2 Core Design Principle

"Hermes asks: 'What should I do?' The harness answers. Hermes executes. Hermes reports back. Loop."

The harness never touches files, never manages API keys, never configures gateways. It thinks. Hermes does. This clean separation is the foundation of H3.

§4 Protocol Design

H3 defines 3 REST endpoints (gRPC available for high-throughput use cases). The OpenAPI 3.1 specification lives at get-h3/protocol and is the single source of truth from which all SDK types are generated.

MethodEndpointPurpose
GET/v1/healthHealth check — status, version, capabilities, latency
POST/v1/processCore: New user message → harness returns a Decision
POST/v1/resultCore: Execution outcome → harness returns next Decision

What Hermes sends to the harness

Every /v1/process request carries the full context the harness needs to reason:

FieldDescription
session_idStable session identifier across turns
messageUser message — content, attachments, timestamp
identityWho sent it — platform, user ID, chat ID, thread ID
context.historyLast N messages (configurable, default 20)
context.toolsAvailable Hermes tools with full JSON Schema params
context.modelsAvailable LLMs with pricing, context window, capabilities
context.memoryDuckBrain memory relevant to this session
context.skillsLoaded skill names
context.configSession config — max iterations, timeout, temperature
context.session_stateRunning counters — turn count, tool calls, cost so far

§5 Decision Types

Every harness response is exactly one Decision. Hermes routes it, executes it, and reports back to the harness via /v1/result.

tool_call
Execute a Hermes tool — terminal, web search, file read/write, browser. Harness specifies tool name + params. Hermes executes in sandbox.
llm_call
Route a prompt to an LLM. Harness picks model, sets system prompt + temperature. Hermes handles provider routing and rate limits.
text
Send markdown-formatted text to the user. finished: true = final message; false = expect next result.
wait
Pause for external signal. Optional poll_endpoint — Hermes polls every 15s, resumes on success.
delegate
Spawn a sub-agent. Harness defines task + context + optional model. Hermes manages the sub-agent lifecycle.
end
Terminate session. Reasons: task_complete, user_requested, error, timeout, rate_limited, cancelled.

Every SDK Gives You

  • Protocol types — structs/dataclasses/interfaces for all request/response types
  • HTTP server — quick-start handler that marshals/unmarshals the protocol
  • Base Harness class — abstract class with on_process() and on_result()
  • Validation — request + Decision validation before sending to Hermes
  • Test helpers — Mock Hermes client for unit testing harness logic

Example: Minimal Echo Harness (Go — ~30 lines)

// Implement these methods. The SDK handles HTTP, marshalling, and validation.
type EchoHarness struct{}

func (e *EchoHarness) OnProcess(req *protocol.ProcessRequest) (*protocol.Decision, error) {
    return &protocol.Decision{
        Decision:   protocol.DecisionText,
        DecisionID: "echo-001",
        Text: &protocol.TextResp{
            Content:  fmt.Sprintf("You said: %s", req.Message.Content),
            Finished: true,
        },
    }, nil
}

func (e *EchoHarness) OnResult(req *protocol.ResultRequest) (*protocol.Decision, error) {
    return &protocol.Decision{Decision: protocol.DecisionEnd, ...}, nil
}

func (e *EchoHarness) Health() *protocol.HealthResponse {
    return &protocol.HealthResponse{Status: "ok", Version: "1.0.0", ProtocolVersion: "1.0", ...}
}

// One line to serve:
// http.ListenAndServe(":9191", harness.NewHTTPServer(&EchoHarness{}))
Zero → working harness in under 5 minutes. Implement three methods, run h3-test, done.

E2E Region-Style Testing

Each category is a complete functional region. The test battery doesn't test individual functions — it tests that the harness handles the entire user experience correctly, from health check through full conversation to termination.

§8 Security Model

Tool sandboxingHarness CANNOT bypass tool execution sandboxing
Audit trailAll tool execution is logged and attributed to the harness session
AuthorizationUser identity passed in context.identity — harness decides policy

Error Modes & Resilience

Failure ModeBehavior
Harness unreachable at startupLog warning, route all sessions to Hermes native agent
Harness timeout mid-sessionRetry 3× with exponential backoff (1s, 2s, 4s)
Malformed DecisionLog error, return error to user, terminate session
Harness hangs (no response)Timeout at timeout_ms, terminate session
Health check fails 3× consecutiveRemove from pool, re-add on next success (30s polling)

§11 Key Differentiators

🆚 vs. LangChain / LlamaIndex

  • Those are frameworks. H3 is a protocol
  • No vendor lock-in — harnesses work with any platform
  • H3 harnesses don't include platform code — pure intelligence
  • Works with any language; not Python-only

🆚 vs. Custom Agent APIs

  • Every platform invents its own. H3 is one standard
  • 43-test compliance suite — no guessing if it works
  • SDKs handle transport; devs write OnProcess() + OnResult()
  • OpenAPI-spec-driven — codegen for any language
Bottom line: H3 is the missing standard for AI agent interoperability. It's thin enough to implement in an afternoon, rigorous enough to validate with a 43-test compliance suite, and open enough to work with any platform, any language, and any agent architecture. The harness thinks. The platform executes. That's the whole idea.
