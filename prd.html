H3 — Hermes Harness Hooks

A two-endpoint protocol that decouples Hermes Core from agent harnesses. Any harness implementing the H3 contract gains access to every Hermes capability — messaging, tools, models, credentials, cron, skills, MCP, and delegation — without touching platform code.

The Problem & Solution

🔒 The Problem

Today, agent harnesses are tightly coupled to their platform. Changing the platform means rewriting the harness. Changing the harness means forking platform code. Every new language or framework requires a custom integration. Harness developers spend 80% of their time on platform plumbing — auth, tool sandboxing, gateway routing, credential management — instead of agent logic.

🔑 The Solution

H3 defines a thin contract between Hermes Core and any harness. The harness implements three endpoints. Hermes handles everything else. The harness thinks — reasoning, planning, memory, prompt engineering. Hermes does — messaging, tool execution, model routing, credential management, scheduling.

Result: Write a harness once in any language. It works with every Hermes feature forever.

System Architecture

H3 sits as a shim layer inside Hermes Core, translating between the platform's internal systems and any harness's REST or gRPC endpoint. The harness never touches files, keys, or gateways.

API Surface

Six REST endpoints. One gRPC service. The harness must implement three core endpoints; the rest are optional for session management.

MethodEndpointPurposeRequired
GET /v1/health Health check — returns status, version, capabilities YES
POST /v1/process New message arrives → harness returns a Decision YES
POST /v1/result Execution result → harness returns next Decision YES
POST /v1/cancel Cancel in-flight operation (user /stop or new message) Recommended
GET /v1/sessions/:id Session metadata — turn count, status, active decision Optional
DELETE /v1/sessions/:id Force-terminate session (/new command) Optional

Decision Types

The harness returns exactly one Decision per response. Six decision types cover every agent action — from tool execution to sub-agent delegation.

tool_call

Execute a Hermes tool — terminal, web_search, patching, file I/O. Harness specifies tool name, parameters, and optional reasoning. Hermes executes in a sandbox and returns the result via /v1/result.

  • Tool names matched against context.tools
  • Params validated against tool's JSON Schema
  • All execution logged per-session

llm_call

Route a prompt to an LLM. Harness chooses model from context.models, provides system prompt and messages. Hermes handles provider routing, fallback, and cost tracking.

  • Model selection with pricing visibility
  • Configurable temperature and max_tokens
  • Streaming via SSE (REST) or native (gRPC)

text

Send a Markdown response to the user. Set finished: false to continue the loop (expect more decisions). Set finished: true to end the exchange.

  • Full Markdown rendering on all platforms
  • Partial responses with finished: false
  • Media attachments supported

wait

Pause for an external signal — CI completion, human approval, API callback. Harness specifies reason, optional duration, and optional poll endpoint.

  • Duration capped at session timeout
  • Poll endpoint checked every 15 seconds
  • Resumes with /v1/process on completion

delegate

Spawn a sub-agent with a task, context, and optional model selection. Hermes manages the sub-agent lifecycle. Harness receives the summary via /v1/result.

  • Task + context passed to sub-agent
  • Model/provider override supported
  • Cost tracked per delegation

end

Terminate the session. Six end reasons: task_complete, user_requested, error, timeout, rate_limited, cancelled. Optional summary sent to user.

  • Clean session teardown
  • Error relayed to user if applicable
  • Cost and turn count preserved

Session Lifecycle

Every user message triggers a loop between Hermes and the harness. The harness drives every decision; Hermes executes and reports back.

Hermes Gateway
User message arrives (Telegram/Discord/Slack/Web). Authenticate, rate-limit, resolve session.
H3 Shim Loader
Look up session → harness mapping from config. Health-check harness. Fallback to native if unreachable.
H3 Shim → Harness
POST /v1/process with session_id, message, history, tools, models, memory, config. Harness thinks → returns Decision.
Decision Router
Hermes reads the Decision: executes tool / routes to LLM / sends text / waits / spawns sub-agent / terminates.
H3 Shim → Harness
POST /v1/result with execution output. Harness processes result → returns next Decision. Loop until end.
User
Receives response on their platform. Session sleeps until next message.

Multi-Language SDKs

All three SDKs are generated from the single OpenAPI 3.1 spec in get-h3/protocol — the single source of truth. Each SDK provides protocol types, HTTP server scaffolding, a base harness class/interface, and test helpers.

Go SDK

github.com/get-h3/sdk-go

  • Struct types with JSON tags from protocol
  • Harness interface: OnProcess() + OnResult()
  • Built-in HTTP handler + middleware (logging, recovery, timeout)
  • MockHermes test bed with assertion helpers
  • Examples: minimal, echo, consensus reference

Python SDK

get-h3/sdk-python (PyPI)

  • Pydantic models from JSON Schema
  • BaseHarness ABC with async methods
  • FastAPI router for quick-start server
  • MockHermes + pytest fixtures
  • Examples: minimal, echo

TypeScript SDK

get-h3/sdk-typescript (npm)

  • Zod schemas for runtime validation
  • Harness interface
  • Hono router for quick-start server
  • MockHermes + vitest helpers
  • Examples: minimal, echo

Design Principles

🧠 Harness Thinks, Hermes Does

Hermes asks: "What should I do?" The harness answers. Hermes executes. Hermes reports back. The harness never touches files, manages API keys, or configures gateways. It thinks. Hermes does.

🔌 Language-Agnostic

REST + JSON means any language works. gRPC for high-frequency loops. No Hermes-specific SDK required — a curl script is a valid harness if it speaks the protocol.

🛡️ Security by Architecture

Harnesses run locally (localhost binding). No network exposure. Harnesses CANNOT access Hermes credential vault. All tool execution is sandboxed and attributed. User identity is passed via context — the harness decides authorization.

🧪 Compliance-Driven

The test battery IS the spec. 43 tests. 6 categories. Region-style E2E coverage. CI integration via GitHub Actions. Pre-commit hooks available. Any developer can run h3-test --endpoint URL and get a full report.

Error Handling & Recovery

FailureBehavior
Harness unreachable at startupLog warning, route sessions to native fallback
Harness timeout mid-sessionRetry 3× with exponential backoff (1s, 2s, 4s)
Malformed Decision returnedLog error, return error to user, end session
Harness hangs (no response)Timeout at configured timeout_ms, terminate session
Harness returns end with errorRelay error to user, terminate cleanly
Health check failsRemove from pool; recheck every 30s; restore on first success
