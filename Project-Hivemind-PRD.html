What Project Hivemind Is

Not another agent framework. A protocol — a contract between platform and brain.

🔌 Protocol, Not Platform

Two core endpoints. REST or gRPC. Any language. No platform code changes to add a new harness. The protocol is the integration surface.

🧠 Harness Owns Thinking

Harnesses decide what to do. The platform executes it. The harness never touches files, keys, or gateways. It thinks. The platform does.

✅ Compliance-Gated

A 43-test battery runs against any harness endpoint. Exit code 0 = H3-compliant. No code changes needed in the harness to be tested.

🌐 Polyglot SDKs

First-class SDKs for Go, Python, and TypeScript — all generated from a single OpenAPI 3.1 spec. Mock test beds ship in every language.

🔐 Secure by Design

Harnesses run locally (localhost). No network exposure. Cannot access credentials directly or bypass tool sandboxing. Platform authenticates; harness trusts.

🔄 Graceful Degradation

Harness unreachable? Sessions route to the native agent loop automatically. Health polling re-adds recovered harnesses to the pool.

The Problem & The Solution

The Problem

Every AI agent platform today couples its reasoning loop to its infrastructure. If you want Telegram + tools + model routing + cron + MCP, you either use the platform's built-in agent (one-size-fits-all, slow to evolve) or you build everything yourself — months of infrastructure work before writing a single line of agent logic.

The Solution

H3 defines a thin contract between platform and harness. Implement three endpoints and your custom agent brain gets all platform infrastructure for free. The protocol is the contract; the platform is the platform. They evolve independently.

Design Principle: "The platform asks: 'What should I do?' The harness answers. The platform executes. The platform reports back. Loop."

Protocol Design

Six REST endpoints. One gRPC service. Harnesses must implement three core endpoints; the rest are optional for session management.

MethodEndpointPurposeRequired
GET/v1/healthHealth check — status, version, capabilitiesYES
POST/v1/processNew message arrives → harness returns a DecisionYES
POST/v1/resultExecution result → harness returns next DecisionYES
POST/v1/cancelCancel in-flight operation (user interrupt)Recommended
GET/v1/sessions/:idSession metadata — turn count, status, active decisionOptional
DELETE/v1/sessions/:idForce-terminate sessionOptional

What the platform sends to the harness on /v1/process

FieldDescription

context.historyLast N messages (configurable, default 20)
context.toolsAvailable tools with full JSON Schema params
context.modelsAvailable LLMs with pricing, context window, capabilities

context.skillsLoaded skill names
context.configSession config — max iterations, timeout, temperature
context.session_stateRunning counters — turn count, tool calls, cost so far

Decision Types

Every harness response is exactly one Decision. Six types cover every agent interaction.

tool_call

Execute a Tool

Harness asks the platform to run a sandboxed command, web search, file operation, or any registered tool. Params validated against JSON Schema.

llm_call

Run an LLM Prompt

Harness delegates reasoning to an LLM through the platform's provider routing. Model selection, system prompts, temperature — all harness-controlled.

text

Send Text to User

Markdown-formatted response to the end user. finished: true ends the turn; finished: false continues with more decisions.

wait

Pause for Signal

Harness pauses execution. Platform polls a URL or resumes on timeout. Used for CI pipelines, human approval, external events.

delegate

Spawn a Sub-Agent

Harness spawns a child agent for parallel work. Task, context, model, and provider are all specified. Results flow back to the harness.

end

Terminate Session

Session ends with a reason code (task_complete, user_requested, error, timeout, rate_limited, cancelled) and an optional summary.

Key Design Decisions

01 · REST First, gRPC Optional

REST is the default transport. Every language speaks HTTP. gRPC is available for high-frequency loops (100+ decisions/task) with native bidirectional streaming. Debuggability wins: curl + jq zero-tooling approach for REST.

02 · Session Resolution with Graceful Degradation

Sessions map to harnesses via config (platform → chat → thread fallback chain). Unreachable harnesses fall back to the native agent loop automatically. Health polling at 30s intervals re-adds recovered harnesses.

03 · Harnesses Run Locally — Security by Proximity

Harnesses bind to localhost. No network exposure by default. They never touch the credential vault or bypass tool sandboxing. The platform authenticates the user; the harness trusts the platform.

04 · One Decision Per Response

Harnesses return exactly one decision per API call. No batched decisions. This keeps the loop simple, predictable, and easy to debug. Tool results always flow back through POST /v1/result.

05 · Version Negotiation at Connection Time

The platform sends H3-Protocol-Version: 1.0. Harness responds with its version. Incompatible versions route to native with an error logged. No silent protocol drift.

06 · OpenAPI as Single Source of Truth

All types, endpoints, schemas, and error codes live in one OpenAPI 3.1 document with companion JSON Schema files. SDKs and the shim generate types FROM this spec — never the reverse.

Security Model

  • Harnesses run locally on the host (localhost binding only)
  • No network exposure by default
  • The platform authenticates the user; the harness trusts the platform
  • User identity passed in context.identity — harness decides authorization
  • Harnesses cannot access the credential vault directly
  • Harnesses cannot bypass tool sandboxing
  • All tool execution is logged and attributed to the harness session
  • All endpoints return standardized error shapes with machine-readable error codes
  • 8 error codes: INVALID_REQUEST, INVALID_DECISION, UNKNOWN_TOOL, UNKNOWN_MODEL, SESSION_NOT_FOUND, SESSION_EXPIRED, HARNESS_TIMEOUT, INTERNAL_ERROR
