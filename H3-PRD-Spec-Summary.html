1 Executive Summary

H3 is the protocol layer between the Hermes AI platform and autonomous agent harnesses. It defines a stable, language-agnostic contract that lets anyone build an AI agent brain in any programming language, wire it into the Hermes ecosystem, and immediately gain access to: Telegram / Discord / Slack / Web gateways, a full tool registry with sandboxed execution, model routing across 8+ providers, credential vaults, cron scheduling, MCP server management, plugin system, and multi-platform I/O — without writing a single line of platform code.

H3 is built around one design principle:

"Hermes asks: 'What should I do?' The harness answers. Hermes executes. Hermes reports back. Loop."

The harness thinks — reasoning, planning, memory, iteration strategy, failure recovery, prompt engineering. Hermes does — all infrastructure, I/O, and external integration.

2 The Problem & Solution

Problem

Every AI agent platform today couples its reasoning loop to its infrastructure. If you want Telegram + tools + model routing + cron + MCP, you either use the platform's built-in agent (one-size-fits-all, slow to evolve) or you build everything yourself (months of infrastructure work before you write your first line of agent logic).

Solution

H3 is three endpoints. Implement them, and your custom agent brain gets all of Hermes's infrastructure for free. The protocol is the contract; the platform is the platform. They evolve independently.

Scheduling — cron, skills, persistenceFailure recovery — retry, fallback, escalation

Session Lifecycle

Here's what happens when a user sends a message on Telegram:

1. Telegram message → Hermes Gateway
2. Gateway authenticates, resolves session → H3 Shim Loader
3. Shim → POST /v1/process (session, message, tools, models, history)
4. Harness thinks → returns Decision (tool_call | llm_call | text | wait | delegate | end)
5. Hermes executes the Decision
6. Hermes → POST /v1/result (execution outcome)
7. Harness thinks → returns next Decision
8. Loop until harness returns end

4 Protocol Design

H3 defines six REST endpoints (gRPC also available for high-throughput use cases). The two most important are /v1/process and /v1/result.

MethodEndpointPurpose
GET/v1/healthHealth check — status, version, capabilities
POST/v1/processCore: New user message → harness returns first Decision
POST/v1/resultCore: Execution outcome → harness returns next Decision
POST/v1/cancelCancel in-flight operation (user interrupt)
GET/v1/sessions/:idSession metadata query
DELETE/v1/sessions/:idForce session termination

What Hermes sends to the harness

Every /v1/process request carries the full context the harness needs to reason:

FieldDescription
session_idStable session identifier across turns
messageUser message — content, attachments, timestamp
identityWho sent it — platform, user ID, chat ID, thread ID
context.historyLast N messages (configurable, default 20)
context.toolsAvailable Hermes tools with full JSON Schema params
context.modelsAvailable LLMs with pricing, context window, capabilities
context.memoryDuckBrain memory relevant to this session
context.skillsLoaded skill names
context.configSession config — max iterations, timeout, temperature
context.session_stateRunning counters — turn count, tool calls, cost so far

Error handling

Failure ModeBehavior
Harness unreachable at startupLog warning, route all sessions to Hermes native agent
Harness timeout mid-sessionRetry 3× with exponential backoff (1s, 2s, 4s)
Malformed DecisionReturn error to user, terminate session
Hang (no response)Timeout at timeout_ms, terminate
Health check fails 3× consecutiveRemove from pool, poll every 30s

5 Decision Types

Every harness response is exactly one Decision. Hermes routes it, executes it, and reports back. Here are all six:

tool_call
Execute a Hermes tool — terminal, web search, file read/write, browser, etc. Harness specifies tool name + params. Hermes executes in sandbox.
llm_call
Route a prompt to an LLM. Harness picks model from available list, sets system prompt + temperature. Hermes handles provider routing.
text
Send markdown-formatted text to the user. finished: true = final message; false = expect /v1/result with next Decision.
wait
Pause for external signal. Optional poll_endpoint — Hermes polls every 15s, resumes on success. Capped at session timeout.
delegate
Spawn a sub-agent. Harness defines task + context + optional model. Hermes manages the sub-agent lifecycle end-to-end.
end
Terminate session. Reasons: task_complete, user_requested, error, timeout, rate_limited, cancelled.

6 SDK Libraries

Three first-class SDKs, all generated from a single OpenAPI 3.1 specification (the protocol repo). Each provides protocol types, an HTTP server, a base harness class, request validation, and a mock Hermes test bed.

SDKLanguageKey LibrariesInstall
sdk-go Go net/http, struct tags, encoding/json go get github.com/get-h3/sdk-go
sdk-python Python Pydantic v2, FastAPI, pytest pip install h3-harness-sdk
sdk-typescript TypeScript Zod, Hono, vitest npm install @get-h3/h3-harness-sdk

What every SDK gives you

  • Protocol types — structs/dataclasses/interfaces for all request/response types
  • HTTP server — quick-start handler that marshals/unmarshals the protocol
  • Base Harness class — abstract class with on_process() and on_result()
  • Validation — request + Decision validation before sending to Hermes
  • Test helpers — Mock Hermes client for unit testing harness logic

Example: Go Harness interface

// Implement these four methods. The SDK handles HTTP, marshalling, and validation.
type Harness interface {
    OnProcess(req *ProcessRequest) (*Decision, error)
    OnResult(req *ResultRequest) (*Decision, error)
    OnCancel(req *CancelRequest) error
    Health() *HealthResponse
}

7 Compliance Test Battery

E2E Region-Style Testing

10 Security Model

  • Harnesses run locally on the Hermes host (localhost binding only)
  • No network exposure by default
  • Hermes authenticates the user; the harness trusts Hermes
  • User identity passed in context.identity — harness decides authorization
  • Harnesses cannot access the Hermes credential vault directly
  • Harnesses cannot bypass tool sandboxing
  • All tool execution is logged and attributed to the harness session
  • All six endpoints return standardized error shapes with machine-readable error codes
